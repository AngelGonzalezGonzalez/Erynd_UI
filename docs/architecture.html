ERYND — how the app works behind the scenes

Two pictures. The first is what you have today (running on your own computer). The second is what it becomes as a real SaaS product running on a server you own, serving many paying customers.

You / users Frontend (what you see) Backend (the brain) Database (the memory) Outside services

1 · Today — running on your computer

Everything except the AI itself runs on your laptop. The AI lives in Anthropic's cloud and is reached over the internet.

You
🧑 Web browser
You open localhost:5173 and chat. You click buttons; you read answers.
▼   you type a message
Frontend · React + Vite
🖼️ The interface
All the screens, charts, and panels. It draws what you see and forwards your message to the backend. It holds no secrets.
▼   asks the backend "what should I answer?"
Backend · Express (Node)
⚙️ The brain / API
Receives the message, decides what to do, talks to the database and the AI, and sends a clean answer back. Your secret API key lives only here.
it then talks to two places
Database · SQLite (one file)
🗄️ The memory
A single file on your disk holding the demo data — mentions, warnings, journalists, the action board, chat history. Seeded once with db:reset.
Outside service · over the internet
🤖 Claude Sonnet 4.6
Anthropic's AI. The backend sends it your message; it writes the reply and picks which panel to open. Billed per use via your API key.

2 · Production — your own SaaS server

Same core idea, but hardened to serve many paying customers safely, reliably, and around the clock — all on infrastructure you own. New pieces are marked NEW.

Your customers
🌍 Many users, any browser
They visit app.yourcompany.com — a real web address, not localhost.
▼   secure connection (https)
Front door NEW
🚪 Reverse proxy + HTTPS
The single guarded entrance (e.g. Nginx/Caddy). Handles the padlock/encryption, blocks abuse, and routes traffic to the right place.
routes to your two running apps
Frontend (served as files)
🖼️ The interface
Same React app, pre-built and served fast to everyone.
Backend · several copies NEW
⚙️ The brain (scaled)
Multiple identical copies running at once so many users are served smoothly and one crash doesn't take the app down.
the backend now relies on a fuller support cast
Database UPGRADED
🗄️ PostgreSQL
A professional database (replaces the single file). Keeps every customer's data separate and safe. Built so this swap is a one-line change.
Accounts & billing NEW
🔐 Login + subscriptions
Sign-up, log-in, "who is this and what plan are they on", and payments (e.g. Stripe). Replaces today's single shared token.
Background workers NEW
⏱️ Always-on jobs + queue
The "watching" engine — continuously pulls in mentions and runs alerts on a schedule, separate from live chats so nothing slows down.
and these outside services, over the internet
Outside · AI
🤖 Claude (Anthropic)
The intelligence, exactly as today — or your future Apify-based agent, swappable without touching the rest.
Outside · data sources NEW
📡 Apify / social & news feeds
Where real mentions come from instead of demo data.
Safety net NEW
💾 Backups + monitoring
Automatic backups, error alerts, and health dashboards so you sleep at night.
New in productionWhy you need it as a SaaS
Front door (reverse proxy + HTTPS)A public, encrypted, single entrance with the padlock — required the moment strangers use your app.
Several backend copiesHandles many users at once and stays up if one copy crashes. This is "scaling."
PostgreSQL databaseThe single-file database is perfect for one person, not for hundreds. Postgres is the proven choice — and we built the app so switching is trivial.
Accounts & billingReal sign-up/login, keeping each customer's data private, and charging subscriptions.
Background workers + queueThe monitoring runs continuously on a schedule, not only when someone is chatting.
Real data sources (Apify etc.)Live mentions from social/news replace the demo dataset.
Backups & monitoringProtects against data loss and tells you when something breaks before customers do.
The reassuring part: the shape barely changes. Frontend → Backend → Database + AI is identical in both pictures. Going to production is mostly adding a support cast around the same core — and two of the biggest jumps (swapping the database to Postgres, swapping the AI for an Apify agent) were deliberately designed in from day one as near one-line changes.
"Run on a server I own"
Every box above except the two cloud "outside services" runs on your own machine — a rented server (e.g. Hetzner, a VPS, or your own hardware). You control the data and the code. Only the AI and the live data feeds are rented per-use from outside, because hosting those yourself isn't practical.
What stays the same
The code you have now is the core of the production system — not a throwaway prototype. The frontend, the backend brain, the database design, and the swappable-AI seam all carry straight over. Production work is hardening and surrounding it, not rebuilding it.
